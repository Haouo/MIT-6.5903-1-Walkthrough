## 一句話總結（TL;DR）

深度學習很有效，但它在**運算量與能耗**上代價驚人——而我們倚賴數十年的硬體基礎（吃著摩爾定律與 Dennard scaling 紅利的通用 CPU）已經不再免費提供效能進步。解方是為深度神經網路（Deep Neural Network, DNN）打造**領域專屬（domain-specific / 特化）硬體**。本課程教你如何透過一組精簡的屬性「讀懂並推理」任何一個 DNN 加速器——包括運算的順序、如何切分（partition）、資料如何流動、資料存放在記憶體階層的哪一層，以及如何利用稀疏性（sparsity）與降低精度（reduced precision）。第一講說明**為什麼**這件事重要，引入貫穿整門課的組織框架 **TeAAL「關注點金字塔」（Pyramid of Concerns）**，並建立一個典型加速器範本——其**能耗由資料搬移主導，而非算術運算**。

讀完本講後，你應該能夠：

- 從**運算量成長**與**能耗／成本**兩方面，說明**為什麼深度學習造成了一場硬體危機**。

## 第一章 — 為什麼要為深度學習打造硬體？

> *投影片：L01-2 … L01-17*

### 三項要素，與真正咬人的那一個

現代 AI 建立在三項要素之上：**大數據的可得性（big data availability）**、**GPU 加速（GPU acceleration）**，以及**新的機器學習技術（new ML techniques）**。投影片以驚人的數字量化資料面——每天上傳數億張影像、每分鐘上傳數百小時影片、每小時數 PB（petabyte）的客戶資料。

![AI 要素——大數據、GPU 加速、新 ML 技術](../../assets/L01/L01-p02-ai-ingredients.png)

三者之中，驅動「本課程」的是第二項。正如 Ilya Sutskever 在 ACM 圖靈獎慶祝活動上所說：**「運算一直是深度學習的氧氣（Compute has been the oxygen of deep learning）。」** 所有進展都是用愈來愈龐大的運算量換來的，而那正是硬體架構師所掌控的資源。

### 運算量的爆炸

最關鍵的數字：從 **AlexNet（2012）到 AlphaGo Zero（2017），里程碑模型所需的運算量成長了約 300,000 倍**——這是以 petaflop/s-days 為單位的指數曲線。這並非摩爾定律那種平緩成長，而是遠快於硬體本身的進步速度。

![運算需求成長——從 AlexNet 到 AlphaGo Zero 成長 300,000 倍](../../assets/L01/L01-p09-compute-demand-growth.png)

### 它所耗費的能源與金錢

這份對運算的胃口會以**能源**與**金錢**的形式體現：

- **資料中心（data center）** 在 2022 年消耗了約 3% 的美國／全球電力需求，預估到 **2030 年將達約 8%**。
- **訓練 GPT-3**（96 層、1,750 億參數、需約 3.14×10²³ FLOPs）在**單張 Tesla V100 上要花約 355 年**，以最便宜的雲端 GPU 計價也需**約 460 萬美元**。**GPT-4 估計超過 1 億美元。**
- **推論**雖然單次便宜，卻持續不斷地執行：ChatGPT 的推論成本估計為**每月 150 萬至 800 萬美元**。

### 訓練 vs. 推論——關鍵區別

這個區別會貫穿整門課程：

- **訓練：** *單次迭代代價高、頻率低。*
- **推論：** *單次迭代代價低、頻率高。*

由於推論一直在跑，它主導了部署後的能耗。在 Google（2019–2021），**約 3/5 的 ML 能耗用於推論**；在 Meta，AI 基礎設施的電力大致以 **10:20:70** 分配於實驗、訓練與推論。圍繞此事的財務競逐——Meta 採購 35 萬張 H100 GPU、全產業的 GPU 短缺——正是「為何高效硬體至關重要」的總體經濟背景。

> **為什麼重要：** DNN 的運算需求成長速度遠快於通用硬體所能跟上的步調，而瓶頸日益落在**能源**而非單純的算術能力上。這道鴻溝正是本課程存在的理由。

## 第二章 — 為什麼需要「特化」硬體？

> *投影片：L01-18 … L01-25*

### 免費的午餐結束了

數十年來，程式隨著電晶體微縮而免費變快。讓這件事成立的兩股趨勢，如今都已**大幅趨緩**：

- **摩爾定律（Moore's Law）** ——電晶體密度倍增——已經放慢。
- **Dennard scaling** ——電晶體微縮時功率密度維持不變——大約在 2000 年代中期實質終結。

其後果是：**通用 CPU 已不再靠自己就能明顯變快或變得更省電。** 想在速度與能效上取得大幅改善，就必須打造**特化／領域專屬硬體**。

![摩爾定律與 Dennard scaling 的趨緩](../../assets/L01/L01-p22-moore-dennard-slowdown.png)

### CPU 複雜度的邊際遞減

本講帶過 CPU 管線的演進——**簡單的循序（in-order）→ 亂序（out-of-order）→ 同時多執行緒（SMT）**——以帶出一個重點：每一代都加入了龐大的機制（分支預測、暫存器重新命名、相依性預測、聰明的快取替換、執行緒選擇）來從通用程式碼中榨出更多效能。這些設計很巧妙，卻在「與真正有用的算術無關的開銷」上耗費了**大量**面積與能量。對於 DNN 這種規律、算術密集的工作負載而言，這些機制大多是浪費——而這恰好是特化加速器可以切入的破口。

### 為什麼要把運算搬到裝置端（on-device）

除了純粹的效率，把 DNN 跑在**裝置上**而非雲端，還有系統層級的理由：

- **隱私（privacy）** ——資料完全不離開裝置。
- **延遲（latency）** ——不需要往返資料中心。
- **通訊（communication）** ——不依賴頻寬／連線。

**自駕車**是極端案例：相機與雷達每 30 秒產生約 6 GB 資料，原型車的運算耗電約 **2,500 W**（高到需要水冷），而一輛自駕車在 10 顆相機上以 60 Hz 跑 10 個 DNN，每小時約做 **2,160 萬次推論**。用本講的話說，它是一座**「裝在輪子上的資料中心」（data center on wheels）**——卻只有極小的功率與散熱預算。

> **為什麼重要：** 通用處理器太耗電（常常 >10 W，但預算只有 <1 W），又進步太慢。要達到所需的能效，唯一的路就是針對 DNN 領域特化的硬體。

## 第三章 — 加速器的設計空間

> *投影片：L01-26 … L01-35*

### 每一個加速器都獨一無二

本講展示了一整排 2017 年以來的真實張量（tensor）加速器——Eyeriss、ExTensor、Eyeriss v2、Gamma、SCNN、spZip、ISOSceles、RAELLA、Highlight、Trapezoid、FuseMax——而它們長得完全不一樣。

![每個加速器都獨一無二——設計集錦](../../assets/L01/L01-p26-every-accelerator-unique.png)

這帶出一個教學上的難題：如果每個設計都獨一無二，你要怎麼在不死背的情況下**理解**一個新設計？答案是擁有一套**關注點框架（framework of concerns）**，讓任何加速器都能被分解到其中。

### TeAAL 關注點金字塔（Pyramid of Concerns）

這是**本講最重要的一張圖**，也是貫穿整門課的組織骨架。它把「設計／分析加速器」的關注點拆解成層層相疊的決策，並由 **Architecture（架構）** 作為約束（constraints）的外殼，限制其下的每一項選擇：

![TeAAL 關注點金字塔——Compute、Mapping、Format、Binding，受 Architecture 約束](../../assets/L01/L01-p28-teaal-pyramid.png)

如何閱讀這座金字塔：

- **Architecture（架構，左側欄）** ——硬體結構本身：PE、記憶體階層、晶片上網路。它對其下所有層級施加**約束（constraints）**。
- **Compute（運算，頂層）** —— *算什麼*：運算／演算法（例如要評估的 Einsum）。
- **Mapping（映射）** —— *如何*把該運算排程到硬體上：迴圈順序（資料流 dataflow）、分塊／切分（tiling/partitioning）、平行性，以及資料放在何處。
- **Format（格式）** —— *資料如何表示*，尤其是壓縮／稀疏性的表示。
- **Binding（綁定）** —— *如何*把抽象的映射綁定到具體硬體資源（哪一個 PE、哪一塊緩衝區、在何時）。

往金字塔上方移動代表**更細緻、更精簡的決策**；往下移動則代表更具體、資源層級的綁定。正是這種「關注點分離（separation of concerns）」讓你能在對等的基礎上比較兩個天差地別的加速器：你對每一個都問同樣的問題。

### 案例研究：FuseMax——四根槓桿，四項勝果

本講以 FuseMax（注意力機制加速，MICRO 2024）說明金字塔各層是**可以各自拉動的獨立槓桿**。每一項增強都針對不同層、帶來不同性質的改善：

![FuseMax 的增強對應到金字塔各層](../../assets/L01/L01-p29-fusemax-enhancements.png)

- **改進運算（improved computation）** → 減少資料搬移（Compute 層的改動）。
- **改變架構（changed architecture）** → 使更好的映射成為可能（Architecture 層的改動）。
- **更好的映射（better mapping）** → 利用新架構所暴露出來的能力。
- **改進綁定（improved binding）** → 提升資源利用率。

接著本講透過一連串 **PE 利用率（PE utilization）** 的投影片（基準 → 增強運算 → 改進架構／映射 → 改進綁定）視覺化其成效，最後收斂到一個在注意力運算上實測得到的**加速比（speedup）**。重點不在那些具體數字，而在**方法**：你一次只針對金字塔的一層去推理一個加速器。

> **為什麼重要：** 金字塔把「每個加速器都獨一無二」轉化為一個可處理的問題。本課程其餘部分本質上就是逐層深入每一層——而實驗（lab）與專題（project）會要求你親手去拉動這些槓桿。

而真正的交付物是這份**課程目標（course objective）**：到課程結束時，你應該能用一組固定詞彙來刻畫*任何*新設計——

- 運算的**順序（order）**
- 運算的**切分（partitioning）**
- 運算的**資料流（flow of data）**
- 儲存階層中的**資料搬移（data movement）**
- **資料屬性專屬的最佳化（data-attribute-specific optimizations）**（稀疏性、精度）
- **演算法／硬體協同設計（co-design）**

請把這份清單收好。它就是閱讀後續每一講都該戴上的那副鏡片。

### 典型加速器範本——以及那個主宰一切的事實

本課程幾乎每個加速器都是同一個範本的變體：**DRAM → 全域緩衝區（Global Buffer）→ 一陣列的處理單元（Processing Element, PE）→ 每個 ALU 旁的微型暫存器檔（Register File, RF）。** 它是一種**空間架構（spatial architecture）**，把小型（<1 kB）、低成本的記憶體擺在靠近運算之處，並以橫跨 200–1000 個 PE 的晶片上網路（Network-on-Chip, NoC）相連。

![典型 DNN 加速器與正規化能耗階層](../../assets/L01/L01-p43-accelerator-energy-cost.png)

關鍵洞見是該投影片右側的**正規化能耗階層（normalized energy-cost hierarchy）**（在商用 65 nm 製程上量測），其中以單次乘加（MAC）的*資料存取*能耗對 ALU 做正規化：

| 資料來源 | 相對能耗 |
|---|---|
| 暫存器檔（RF）／PE 內部 | **1×（基準）** |
| 經 NoC 的鄰近 PE | **2×** |
| 全域緩衝區（100–500 kB） | **6×** |
| **DRAM** | **200×** |

**從 DRAM 讀一筆資料，耗能約是做一次算術的 200 倍。** 這個單一事實重新框定了整個設計問題：**主導能耗的是資料搬移，而非運算。** 本課程其餘所有技術——資料流、分塊、駐留性（stationarity）、融合（fusion）、稀疏性、降低精度、記憶體內運算（compute-in-memory）——本質上都是一種策略：**讓資料靠近、少搬一點。**

### 你將如何探索設計空間

- 你會操作的**設計選擇**：PE 陣列（數量、NoC 拓樸）、記憶體階層（層數、各層容量、資料排佈）、排程／映射（資料流、分塊、平行性、融合）、稀疏性處理（閘控 gating、跳過 skipping、壓縮格式）、以及實作技術（RRAM、光學、超導體）。
- **屋頂線模型（roofline model）** 被引入作為系統化的評估方法，用以衡量每一項架構決策如何影響特定工作負載的吞吐量——它是一連串從理論峰值效能逐步「收緊（tightening）」到現實可達效能的步驟（受限於有限的 PE 數、固定的陣列維度、有限的儲存空間與有限的頻寬）。

![屋頂線模型——逐步收緊](../../assets/L01/L01-p45-roofline-model.png)

- **用建模而非 RTL。** 由於用 RTL 逐一實作每個設計太慢，本課程改用**架構建模工具（Accelergy 與 AccelForge）**來快速探索設計空間。（本課不含 GPU 程式設計或 FPGA 實作。）

### 課務（簡述）

這些機制因不屬於智識內容而從簡：課程使用 **Python + PyTorch**；教科書是 Sze 與 Emer 所著的 *Efficient Processing of Deep Neural Networks*（在 MIT 網路內免費下載）。評分為 **50% 實驗／50% 期末專題**。實驗依金字塔逐層推進：Lab 1（Einsum／工作負載分析）→ Lab 2–3（硬體設計與映射）→ Lab 4（稀疏性）→ Lab 5（記憶體內運算）。專題則把這些工具應用到真實加速器的研究上。（教職員、時程、評分、遲交政策與先修要求請見投影片 L01-40 … L01-53。）

> **為什麼重要：** 這門課的承諾是一項*可遷移的技能*——不是熟悉某一顆晶片，而是一套能讀懂任何晶片的方法。而能耗階層，就是讓多數設計決策變得合理的那顆北極星。

## 獨立學習指南（Standalone Study Guide）

### 進入下一講前必須掌握

- 說明為什麼 DNN 硬體的限制常常來自資料搬移，而不只是算術吞吐量。
- 用 TeAAL 金字塔區分模型、表示法、映射、架構與技術層的設計決策。
- 讀懂 Roofline 圖，判斷加速器是受算力限制還是受頻寬限制。
- 說明為什麼不存在單一「最佳加速器」：工作負載形狀、資料屬性與映射選擇都會改變最佳點。

### 自我檢核問題

1. 如果 DRAM 能耗遠大於 MAC 能耗，為什麼有些加速器增加 MAC 數反而可能降低總能耗？
2. 稀疏性與精度位於金字塔的哪一層？為什麼它們是協同設計問題，而不是純架構技巧？
3. 在決定一個工作負載該放到 GPU、TPU 類 systolic array，或客製稀疏加速器之前，你需要哪些資訊？

### 練習

1. 選一個真實 DNN 工作負載，判斷它最可能受限於算力、記憶體頻寬、片上儲存，或控制不規則性，並用本講詞彙說明理由。
2. 為 FuseMax 畫一張一頁版金字塔，標出哪些決策屬於演算法、哪些屬於映射、哪些屬於架構。

### 常見誤區

- 把 TOPS 當成完整效能指標。它忽略記憶體流量、利用率與工作負載形狀。
- 把「特化」誤解為「完全不彈性」。本講強調的是針對穩定工作負載結構做有範圍的特化。
- 把金字塔讀成單向瀑布流程。實際設計中，各層決策會互相回饋。

## 關鍵詞彙（Key Terms）

| 詞彙 | 說明 |
|---|---|
| **DNN（Deep Neural Network，深度神經網路）** | 本硬體所針對的工作負載類別。 |
| **訓練 vs. 推論（Training vs. Inference）** | 訓練＝代價高／頻率低；推論＝代價低／頻率高（且主導部署後能耗）。 |
| **摩爾定律（Moore's Law）** | 電晶體密度倍增的趨勢；現已趨緩。 |
| **Dennard scaling** | 電晶體微縮時功率密度維持不變；約於 2000 年代中期實質終結。 |
| **領域專屬／特化硬體（domain-specific / specialized hardware）** | 為某工作負載領域量身打造、以取得 CPU 無法企及之速度／能效的硬體。 |
| **加速器（Accelerator）** | 用於 DNN 運算的特化硬體區塊。 |
| **PE（Processing Element，處理單元）** | 基本運算單元（一個 ALU 加上小型本地記憶體），可排成陣列。 |
| **MAC（Multiply-Accumulate，乘加）** | DNN 最基本的算術運算。 |
| **RF（Register File，暫存器檔）** | 每個 ALU 旁的微型（<1 kB）記憶體；亦稱本地緩衝區（local buffer）。 |
| **全域緩衝區（Global Buffer）** | PE 之間共享的晶片上 SRAM（100–500 kB）。 |
| **NoC（Network-on-Chip，晶片上網路）** | 連接 PE 陣列（200–1000 個 PE）的互連網路。 |
| **TeAAL 關注點金字塔** | 本課框架：Architecture 約束 Compute → Mapping → Format → Binding。 |
| **Mapping（映射）** | 如何把運算排程到硬體上（資料流、分塊、平行性、放置）。 |
| **Format（格式）** | 資料如何表示，尤其是稀疏／壓縮。 |
| **Binding（綁定）** | 把抽象映射指派到具體硬體資源。 |
| **Einsum** | 本課用來表達 DNN 運算的記法（後續講次引入）。 |
| **屋頂線模型（Roofline model）** | 用以界定可達吞吐量相對於運算強度（compute intensity）上界的方法。 |
| **Accelergy / AccelForge** | 實驗中使用的架構建模工具。 |
| **正規化能耗（Normalized energy cost）** | 在階層各層存取資料的相對能耗（RF 1× … DRAM 200×）。 |

## 重點回顧（Takeaways）

- 深度學習的運算需求在五年內成長了**約 300,000 倍**；愈來愈吃緊的限制是**能源**。
- **摩爾定律／Dennard scaling 已趨緩**，通用 CPU 無法提供所需的增益——**特化硬體**才是解答。
- 即使訓練單次代價更高，**推論卻主導了部署後能耗**（例如 Google 約 3/5、Meta 約 70% 的 AI 電力）。
- **TeAAL 關注點金字塔**（Architecture → Compute、Mapping、Format、Binding）是推理*任何*加速器的框架。
- 典型加速器是 **DRAM → 全域緩衝區 → PE 陣列 → RF/ALU**，而**資料搬移主導能耗**：DRAM 存取約為一次 ALU 運算的 **200 倍**。
- 課程目標是一套**可遷移的方法**：以順序、切分、資料流、記憶體搬移、資料專屬最佳化、協同設計與彈性來刻畫任何設計。
